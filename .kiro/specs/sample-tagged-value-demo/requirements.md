# Requirements Document

## Introduction

This feature updates the existing Maven and Gradle sample projects to demonstrate the new `@TaggedValue` annotation introduced in rawit `1.1.0-rc-1`. Both samples currently showcase `@Invoker`, `@Constructor`, and `@Getter` usage with version `1.0.0`. This update bumps the rawit dependency to `1.1.0-rc-1`, adds new model classes that define and use tag annotations (`@UserId`, `@FirstName`, `@LastName`), and adds JUnit 5 tests that exercise the generated builder chains with tagged parameters. Both samples continue to share identical Java source files.

## Glossary

- **Maven_Sample**: The sample project located at `samples/maven-sample/` that uses Apache Maven as its build tool
- **Gradle_Sample**: The sample project located at `samples/gradle-sample/` that uses Gradle (Groovy DSL) as its build tool
- **Rawit_Dependency**: The published Maven Central artifact `io.github.projectrawit:rawit:1.1.0-rc-1`
- **TaggedValue_Annotation**: The `@TaggedValue` meta-annotation from the rawit library that marks custom annotations as value type tags
- **Tag_Annotation**: A user-defined annotation that is itself annotated with `@TaggedValue`, such as `@UserId`, `@FirstName`, or `@LastName`
- **Strict_Tag**: A Tag_Annotation declared with `@TaggedValue(strict = true)`, which causes warnings on tagged-to-untagged and untagged-to-tagged assignments (except for literals and constants)
- **Lax_Tag**: A Tag_Annotation declared with `@TaggedValue(strict = false)` (the default), which only warns on tag mismatches between different tags
- **Tagged_Record**: A record class annotated with `@Constructor` whose record components carry Tag_Annotations
- **Staged_Call_Chain**: The fluent API generated by rawit where each parameter becomes a chained method call, ending with `.invoke()` or `.construct()`
- **Sample_Source_Files**: The Java source files under `src/` in each sample project that are shared identically between Maven_Sample and Gradle_Sample

## Requirements

### Requirement 1: Update Rawit Dependency Version

**User Story:** As a developer evaluating rawit, I want the sample projects to reference the latest rawit version, so that I can see the newest features including `@TaggedValue`.

#### Acceptance Criteria

1. THE Maven_Sample SHALL declare a dependency on Rawit_Dependency with groupId `io.github.projectrawit`, artifactId `rawit`, and version `1.1.0-rc-1` in `samples/maven-sample/pom.xml`
2. THE Gradle_Sample SHALL declare Rawit_Dependency as both an `annotationProcessor` and `compileOnly` dependency with version `1.1.0-rc-1` in `samples/gradle-sample/build.gradle`

### Requirement 2: Define Tag Annotation Declarations

**User Story:** As a developer, I want to see how to define custom tag annotations using `@TaggedValue`, so that I can create my own value type tags.

#### Acceptance Criteria

1. THE Maven_Sample SHALL contain a Tag_Annotation declaration `@UserId` annotated with `@TaggedValue(strict = true)` in the `com.example.model` package
2. THE Maven_Sample SHALL contain a Tag_Annotation declaration `@FirstName` annotated with `@TaggedValue` (lax mode, the default) in the `com.example.model` package
3. THE Maven_Sample SHALL contain a Tag_Annotation declaration `@LastName` annotated with `@TaggedValue` (lax mode, the default) in the `com.example.model` package
4. THE Gradle_Sample SHALL contain Tag_Annotation declarations identical to those in the Maven_Sample
5. WHEN a Tag_Annotation declaration uses `@TaggedValue(strict = true)`, THE declaration SHALL include a code comment explaining that strict mode warns on tagged-to-untagged and untagged-to-tagged assignments
6. WHEN a Tag_Annotation declaration uses `@TaggedValue` without specifying `strict`, THE declaration SHALL include a code comment explaining that lax mode only warns on tag mismatches

### Requirement 3: Define a Tagged Record Using @Constructor

**User Story:** As a developer, I want to see a record class that combines `@Constructor` with tag annotations on its components, so that I can understand how tagged values integrate with rawit's builder chain generation.

#### Acceptance Criteria

1. THE Maven_Sample SHALL contain a record class named `TaggedUser` in the `com.example.model` package annotated with `@Constructor`
2. THE `TaggedUser` record SHALL have a `@UserId long userId` component, a `@FirstName String firstName` component, and a `@LastName String lastName` component
3. THE Gradle_Sample SHALL contain a `TaggedUser` record identical to the one in the Maven_Sample
4. THE `TaggedUser` record SHALL demonstrate that Tag_Annotations propagate onto the generated builder chain stage method parameters

### Requirement 4: Demonstrate Tagged Values with @Invoker

**User Story:** As a developer, I want to see a class that combines `@Invoker` with tag annotations on method parameters, so that I can understand how tagged values integrate with rawit's invoker chain generation.

#### Acceptance Criteria

1. THE Maven_Sample SHALL contain a class named `UserService` in the `com.example.model` package with at least one method annotated with `@Invoker` whose parameters carry Tag_Annotations
2. THE `UserService` class SHALL include a method that accepts `@FirstName String` and `@LastName String` parameters, demonstrating tagged parameters on an instance method
3. THE Gradle_Sample SHALL contain a `UserService` class identical to the one in the Maven_Sample

### Requirement 5: Demonstrate Tagged Values with @Getter

**User Story:** As a developer, I want to see a class that combines `@Getter` with tag annotations on fields, so that I can understand how tagged values work with rawit's getter generation.

#### Acceptance Criteria

1. THE `TaggedUser` record or the `UserService` class SHALL include fields annotated with both `@Getter` and a Tag_Annotation, demonstrating that getter-generated accessor methods return tagged values
2. THE Maven_Sample and Gradle_Sample SHALL contain identical source files demonstrating this combination

### Requirement 6: JUnit 5 Tests for Tagged Builder Chain with Literals

**User Story:** As a developer, I want to see tests that construct a `TaggedUser` using the generated builder chain with literal values, so that I can confirm literals are exempt from strict-mode warnings.

#### Acceptance Criteria

1. THE Maven_Sample SHALL contain a JUnit 5 test method that constructs a `TaggedUser` via the Staged_Call_Chain using literal values for all three parameters
2. THE test SHALL verify that the constructed `TaggedUser` contains the correct `userId`, `firstName`, and `lastName` values
3. THE Gradle_Sample SHALL contain a test identical to the one in the Maven_Sample

### Requirement 7: JUnit 5 Tests for Tagged Builder Chain with Tagged Variables

**User Story:** As a developer, I want to see tests that pass correctly-tagged variables into the builder chain, so that I can confirm same-tag assignments produce no warnings.

#### Acceptance Criteria

1. THE Maven_Sample SHALL contain a JUnit 5 test method that declares local variables annotated with `@UserId`, `@FirstName`, and `@LastName`, then passes them to the `TaggedUser` Staged_Call_Chain
2. THE test SHALL verify that the constructed `TaggedUser` contains the correct values from the tagged local variables
3. THE Gradle_Sample SHALL contain a test identical to the one in the Maven_Sample

### Requirement 8: JUnit 5 Tests for Tagged Invoker Chain

**User Story:** As a developer, I want to see tests that exercise the `@Invoker`-generated chain with tagged parameters, so that I can confirm tagged values work with invoker chains.

#### Acceptance Criteria

1. THE Maven_Sample SHALL contain a JUnit 5 test method that invokes the `UserService` method via the Staged_Call_Chain, passing tagged local variables as arguments
2. THE test SHALL verify that the invoked method returns the correct result
3. THE Gradle_Sample SHALL contain a test identical to the one in the Maven_Sample

### Requirement 9: JUnit 5 Tests for Tagged Getter

**User Story:** As a developer, I want to see tests that exercise getter-generated accessors on tagged fields, so that I can confirm tagged values work with getter generation.

#### Acceptance Criteria

1. THE Maven_Sample SHALL contain a JUnit 5 test method that calls a getter-generated accessor on a tagged field and verifies the returned value
2. THE Gradle_Sample SHALL contain a test identical to the one in the Maven_Sample

### Requirement 10: JUnit 5 Test for Accessor Methods on Tagged Record

**User Story:** As a developer, I want to see tests that call the accessor methods on a `TaggedUser` instance, so that I can confirm the record's generated accessors work correctly alongside the tagged builder chain.

#### Acceptance Criteria

1. THE Maven_Sample SHALL contain a JUnit 5 test method that constructs a `TaggedUser` and verifies each accessor method (`userId()`, `firstName()`, `lastName()`) returns the expected value
2. THE Gradle_Sample SHALL contain a test identical to the one in the Maven_Sample

### Requirement 11: Preserve Existing Sample Functionality

**User Story:** As a developer, I want the existing `@Invoker`, `@Constructor`, `@Getter`, and record constructor demonstrations to remain intact after the update, so that the samples continue to showcase all rawit features.

#### Acceptance Criteria

1. THE Maven_Sample SHALL retain the existing `Calculator.java`, `Point.java`, `Coord.java`, and `User.java` model classes without modification
2. THE Maven_Sample SHALL retain all existing test methods in `RawitSampleTest.java` without modification
3. THE Gradle_Sample SHALL retain the same existing model classes and test methods as the Maven_Sample

### Requirement 12: Source File Identity Across Samples

**User Story:** As a maintainer, I want both sample projects to share identical Java source files, so that maintenance effort is minimized and developers can compare build configurations without source code differences.

#### Acceptance Criteria

1. FOR ALL Java source files that exist in both `samples/maven-sample/src/` and `samples/gradle-sample/src/` at the same relative path, THE file content SHALL be byte-for-byte identical
2. THE Maven_Sample and Gradle_Sample SHALL contain the same set of Java source files under `src/`

### Requirement 13: Sample Projects Remain Self-Contained

**User Story:** As a developer, I want each sample project to remain independently buildable after the update, so that I can clone and run either sample without needing the parent project.

#### Acceptance Criteria

1. THE Maven_Sample SHALL be buildable by running `mvn clean test` from the `samples/maven-sample/` directory, resolving Rawit_Dependency from Maven Central or Maven Local
2. THE Gradle_Sample SHALL be buildable by running `gradle clean test` from the `samples/gradle-sample/` directory, resolving Rawit_Dependency from Maven Central or Maven Local
