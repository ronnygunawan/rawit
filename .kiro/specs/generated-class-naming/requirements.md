# Requirements Document

## Introduction

This feature changes the naming scheme for classes generated by the Rawit annotation processor and relocates them into a `.generated` subpackage. The current scheme causes name collisions when multiple `@Constructor`-annotated classes exist in the same package (e.g., both `Point` and `Coord` generate a top-level class named `Constructor`). The new scheme prefixes generated class names with the enclosing class name and appends a suffix indicating the annotation type, then places all generated classes in a `<pkg>.generated` subpackage. All three code-generation components (InvokerClassSpec, JavaPoetGenerator, BytecodeInjector) must adopt the new naming convention consistently.

## Glossary

- **Processor**: The `RawitAnnotationProcessor` that processes `@Invoker` and `@Constructor` annotations
- **InvokerClassSpec**: The component that builds the `TypeSpec` for a generated class, including determining the class name via `resolveCallerClassName()`
- **JavaPoetGenerator**: The component that writes generated Java source files via JavaPoet, including determining the target package via `resolvePackageName()`
- **BytecodeInjector**: The component that injects parameterless bridge methods into `.class` files using ASM, including determining the generated class binary name via `resolveCallerClassBinaryName()`
- **Caller_Class**: The generated top-level class containing stage interfaces and the terminal method (e.g., `PointConstructor`, `CalculatorAddInvoker`)
- **Generated_Subpackage**: The `.generated` subpackage appended to the enclosing class's package (e.g., `com.example.model.generated`)
- **Enclosing_Class**: The class or record that contains the annotated method or constructor
- **OverloadGroup**: The model record carrying `enclosingClassName` and `groupName` for a group of annotated methods

## Naming Scheme

| Annotation | Enclosing Class | Method/Constructor | Current Generated | New Generated |
|---|---|---|---|---|
| `@Invoker` on method | `Foo` | `bar(int x, int y)` | `<pkg>.Bar` | `<pkg>.generated.FooBarInvoker` |
| `@Invoker` on method | `Calculator` | `add(int x, int y)` | `<pkg>.Add` | `<pkg>.generated.CalculatorAddInvoker` |
| `@Invoker` on method | `Calculator` | `multiply(int a, int b)` | `<pkg>.Multiply` | `<pkg>.generated.CalculatorMultiplyInvoker` |
| `@Invoker` on constructor | `Foo` | `Foo(int x)` | `<pkg>.FooInvoker` | `<pkg>.generated.FooInvoker` |
| `@Constructor` | `Point` | `Point(int x, int y)` | `<pkg>.Constructor` | `<pkg>.generated.PointConstructor` |
| `@Constructor` | `Coord` (record) | `Coord(int lat, int lon)` | `<pkg>.Constructor` | `<pkg>.generated.CoordConstructor` |

## Requirements

### Requirement 1: Rename Generated Classes for @Constructor Annotations

**User Story:** As a developer, I want generated classes for `@Constructor` annotations to include the enclosing class name, so that multiple `@Constructor`-annotated classes in the same package do not collide.

#### Acceptance Criteria

1. WHEN the InvokerClassSpec resolves the Caller_Class name for a `@Constructor`-annotated group, THE InvokerClassSpec SHALL produce a name of the form `<EnclosingSimpleName>Constructor` (e.g., `PointConstructor` for enclosing class `Point`)
2. WHEN the BytecodeInjector resolves the Caller_Class binary name for a `@Constructor`-annotated group, THE BytecodeInjector SHALL produce a binary name of the form `<pkg>/generated/<EnclosingSimpleName>Constructor` (e.g., `com/example/model/generated/PointConstructor`)
3. WHEN two `@Constructor`-annotated classes exist in the same package (e.g., `Point` and `Coord`), THE Processor SHALL generate distinct Caller_Classes named `PointConstructor` and `CoordConstructor` in the Generated_Subpackage

#### Examples

Before (collision):
```
com.example.model.Point  ŌåÆ generates com.example.model.Constructor
com.example.model.Coord  ŌåÆ generates com.example.model.Constructor  ŌåÉ COLLISION
```

After (no collision):
```
com.example.model.Point  ŌåÆ generates com.example.model.generated.PointConstructor
com.example.model.Coord  ŌåÆ generates com.example.model.generated.CoordConstructor
```

### Requirement 2: Rename Generated Classes for @Invoker on Methods

**User Story:** As a developer, I want generated classes for `@Invoker`-annotated methods to include both the enclosing class name and the method name, so that the generated class name is unambiguous.

#### Acceptance Criteria

1. WHEN the InvokerClassSpec resolves the Caller_Class name for an `@Invoker`-annotated non-constructor method, THE InvokerClassSpec SHALL produce a name of the form `<EnclosingSimpleName><PascalCaseMethodName>Invoker` (e.g., `CalculatorAddInvoker` for method `add` in class `Calculator`)
2. WHEN the BytecodeInjector resolves the Caller_Class binary name for an `@Invoker`-annotated non-constructor method, THE BytecodeInjector SHALL produce a binary name of the form `<pkg>/generated/<EnclosingSimpleName><PascalCaseMethodName>Invoker` (e.g., `com/example/model/generated/CalculatorAddInvoker`)

#### Examples

```
Calculator.add(int x, int y)       ŌåÆ generates com.example.model.generated.CalculatorAddInvoker
Calculator.multiply(int a, int b)  ŌåÆ generates com.example.model.generated.CalculatorMultiplyInvoker
```

### Requirement 3: Rename Generated Classes for @Invoker on Constructors

**User Story:** As a developer, I want generated classes for `@Invoker`-annotated constructors to retain the existing `<ClassName>Invoker` pattern but move to the generated subpackage.

#### Acceptance Criteria

1. WHEN the InvokerClassSpec resolves the Caller_Class name for an `@Invoker`-annotated constructor (groupName is `<init>`), THE InvokerClassSpec SHALL produce a name of the form `<EnclosingSimpleName>Invoker` (e.g., `FooInvoker` for class `Foo`)
2. WHEN the BytecodeInjector resolves the Caller_Class binary name for an `@Invoker`-annotated constructor, THE BytecodeInjector SHALL produce a binary name of the form `<pkg>/generated/<EnclosingSimpleName>Invoker` (e.g., `com/example/generated/FooInvoker`)

#### Examples

```
Foo(int x)  with @Invoker  ŌåÆ generates com.example.generated.FooInvoker
```

### Requirement 4: Relocate All Generated Classes to .generated Subpackage

**User Story:** As a developer, I want all generated classes to reside in a `.generated` subpackage, so that generated code is clearly separated from hand-written code.

#### Acceptance Criteria

1. WHEN the JavaPoetGenerator resolves the package name for a generated source file, THE JavaPoetGenerator SHALL append `.generated` to the enclosing class's package (e.g., `com.example.model` becomes `com.example.model.generated`)
2. WHEN the BytecodeInjector resolves the Caller_Class binary name, THE BytecodeInjector SHALL include `generated/` in the package path between the enclosing class's package and the Caller_Class simple name (e.g., `com/example/model/generated/PointConstructor`)
3. WHEN the enclosing class is in the default (empty) package, THE JavaPoetGenerator SHALL use `generated` as the package name
4. WHEN the enclosing class is in the default (empty) package, THE BytecodeInjector SHALL use `generated/` as the package prefix in the binary name

#### Examples

Enclosing class in `com.example.model`:
```
JavaPoetGenerator package: com.example.model.generated
BytecodeInjector binary:   com/example/model/generated/PointConstructor
```

Enclosing class in default package:
```
JavaPoetGenerator package: generated
BytecodeInjector binary:   generated/PointConstructor
```

### Requirement 5: Maintain Consistency Between Source Generation and Bytecode Injection

**User Story:** As a developer, I want the generated source class name and the bytecode-injected bridge method return type to reference the same fully qualified class, so that the compiled code links correctly at runtime.

#### Acceptance Criteria

1. FOR ALL OverloadGroups processed by the Processor, THE Caller_Class name produced by InvokerClassSpec SHALL match the simple name portion of the binary name produced by BytecodeInjector
2. FOR ALL OverloadGroups processed by the Processor, THE package produced by JavaPoetGenerator SHALL match the package portion of the binary name produced by BytecodeInjector (with `/` replaced by `.`)
3. WHEN the BytecodeInjector injects a bridge method, THE BytecodeInjector SHALL use a return type descriptor that references the Caller_Class in the Generated_Subpackage

### Requirement 6: Update Sample Project References

**User Story:** As a developer using the sample projects, I want the sample tests to continue passing after the naming change, so that the samples remain a working reference.

#### Acceptance Criteria

1. WHEN the naming scheme changes, THE sample project test files SHALL reference the generated classes using the new fully qualified names in the Generated_Subpackage
2. WHEN the naming scheme changes, THE sample project test files SHALL continue to use the same staged API call chains (e.g., `Point.constructor().x(10).y(20).construct()`) without modification to the call-site syntax

#### Examples

The call-site syntax remains unchanged because the bridge methods are injected into the original class:
```java
// This call chain does NOT change ŌĆö the bridge method on Point still works
Point p = Point.constructor().x(10).y(20).construct();

// But if tests explicitly import or reference the generated class:
// Before: import com.example.model.Constructor;
// After:  import com.example.model.generated.PointConstructor;
```

### Requirement 7: Update Unit and Property Test Assertions

**User Story:** As a developer maintaining the rawit project, I want all existing unit tests and property tests to pass with the new naming scheme, so that the test suite validates the new behavior.

#### Acceptance Criteria

1. WHEN the naming scheme changes, THE InvokerClassSpecTest SHALL assert that `resolveCallerClassName()` produces names following the new `<EnclosingSimpleName><MethodPascalCase>Invoker` or `<EnclosingSimpleName>Constructor` patterns
2. WHEN the naming scheme changes, THE InvokerClassSpecPropertyTest SHALL assert that generated class names follow the new naming patterns
3. WHEN the naming scheme changes, THE BytecodeInjectorPropertyTest SHALL assert that return type descriptors reference the Caller_Class in the Generated_Subpackage
4. WHEN the naming scheme changes, THE RawitAnnotationProcessorPropertyTest SHALL assert that generated source files use the new class names and package
5. WHEN the naming scheme changes, THE RawitAnnotationProcessorConstructorPropertyTest SHALL load generated classes using the new fully qualified names (e.g., `generated.PointConstructor` instead of `Constructor`)
6. WHEN the naming scheme changes, THE RawitAnnotationProcessorIntegrationTest SHALL assert generated class names and packages follow the new convention
