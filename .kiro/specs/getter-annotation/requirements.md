# Requirements Document

## Introduction

This feature adds a new `@Getter` annotation to the rawit annotation processor project. When applied to a field (static or instance, any visibility), the processor generates a corresponding getter method on the enclosing class. The getter naming follows legacy Lombok conventions for primitive `boolean` fields, while boxed `Boolean` and all other types use the standard `get` prefix.

## Glossary

- **Getter_Processor**: The component within the rawit annotation processor that handles `@Getter`-annotated fields, validates them, computes getter method names, and generates getter method source code or bytecode.
- **Getter_Annotation**: The `@Getter` annotation type, targeting `ElementType.FIELD` with `RetentionPolicy.SOURCE`.
- **Enclosing_Class**: The class or type that contains the annotated field.
- **Getter_Method**: The generated method that returns the value of the annotated field.
- **Primitive_Boolean_Field**: A field whose type is the Java primitive `boolean`.
- **Boxed_Boolean_Field**: A field whose type is `java.lang.Boolean`.
- **Base_Class**: A superclass in an inheritance hierarchy that declares a field.
- **Derived_Class**: A subclass that extends a Base_Class.
- **Field_Hiding**: A situation where a Derived_Class declares a field with the same name as a field in the Base_Class, causing the Base_Class field to be hidden (not overridden) within the Derived_Class scope.
- **Covariant_Return_Type**: A return type in an overriding method that is a subtype of the return type declared in the overridden method. Java permits covariant return types in method overrides (e.g., a base method returning `Number` can be overridden by a method returning `Integer`).
- **Generic_Field**: A field whose declared type includes one or more type parameters (e.g., `List<String>`, `Map<String, Integer>`, `Optional<T>`).
- **Volatile_Field**: A field declared with the `volatile` modifier.
- **Transient_Field**: A field declared with the `transient` modifier.
- **Anonymous_Class**: An unnamed class defined inline via `new SuperType() { ... }` syntax, which cannot be referenced by name.
- **Named_Inner_Class**: A non-static member class declared inside another class with an explicit name.
- **Static_Inner_Class**: A static member class declared inside another class.
- **Local_Class**: A named class declared inside a method body or initializer block.
- **Inherited_Method**: A method declared in a superclass (not generated by the Getter_Processor) that is inherited by a subclass.

## Requirements

### Requirement 1: Annotation Target Restriction

**User Story:** As a developer, I want the `@Getter` annotation to only be applicable to fields, so that misuse on methods, classes, or other elements is caught at compile time.

#### Acceptance Criteria

1. THE Getter_Annotation SHALL target only `ElementType.FIELD`.
2. WHEN a developer applies `@Getter` to a non-field element (method, class, constructor, parameter, etc.), THE Java compiler SHALL emit a compile-time error before the annotation processor runs.

### Requirement 2: Getter Method Generation for Standard Fields

**User Story:** As a developer, I want a getter method generated for each `@Getter`-annotated field, so that I do not have to write boilerplate accessor methods.

#### Acceptance Criteria

1. WHEN a field is annotated with `@Getter`, THE Getter_Processor SHALL generate a Getter_Method on the Enclosing_Class.
2. THE Getter_Method SHALL have the name `get` followed by the field name with the first letter capitalized (camelCase convention). For example, field `firstName` produces getter `getFirstName()`.
3. THE Getter_Method SHALL have a return type matching the type of the annotated field.
4. THE Getter_Method SHALL return the value of the annotated field.
5. THE Getter_Processor SHALL generate getters for fields of any access modifier: `private`, `protected`, `public`, or package-private.

### Requirement 3: Static Field Getter Generation

**User Story:** As a developer, I want the generated getter to be static when the annotated field is static, so that the getter matches the field's binding.

#### Acceptance Criteria

1. WHEN a static field is annotated with `@Getter`, THE Getter_Processor SHALL generate a static Getter_Method.
2. WHEN an instance field is annotated with `@Getter`, THE Getter_Processor SHALL generate an instance (non-static) Getter_Method.

### Requirement 4: Primitive Boolean Getter Naming (Legacy Lombok Behavior)

**User Story:** As a developer, I want primitive `boolean` field getters to follow the Lombok `is`-prefix convention, so that the generated API is consistent with established Java conventions.

#### Acceptance Criteria

1. WHEN a Primitive_Boolean_Field is annotated with `@Getter` and the field name is not prefixed with `is`, THE Getter_Processor SHALL generate a Getter_Method with the name `is` followed by the field name with the first letter capitalized. For example, field `active` produces getter `isActive()`.
2. WHEN a Primitive_Boolean_Field is annotated with `@Getter` and the field name is prefixed with `is` followed by an uppercase letter, THE Getter_Processor SHALL generate a Getter_Method with the same name as the field. For example, field `isActive` produces getter `isActive()`.
3. WHEN a Primitive_Boolean_Field is annotated with `@Getter` and the field name is prefixed with `is` followed by a non-uppercase letter, THE Getter_Processor SHALL generate a Getter_Method with the name `is` followed by the field name with the first letter capitalized. For example, field `isinTimezone` produces getter `isIsinTimezone()`.

### Requirement 5: Boxed Boolean Getter Naming

**User Story:** As a developer, I want boxed `Boolean` field getters to use the standard `get` prefix, so that the naming is consistent with all other non-primitive-boolean types.

#### Acceptance Criteria

1. WHEN a Boxed_Boolean_Field is annotated with `@Getter`, THE Getter_Processor SHALL generate a Getter_Method using the standard `get` prefix naming (same as Requirement 2). For example, field `active` of type `Boolean` produces getter `getActive()`.

### Requirement 6: Getter Name Collision Detection

**User Story:** As a developer, I want the processor to detect getter name collisions, so that I am informed of conflicts at compile time rather than encountering runtime errors.

#### Acceptance Criteria

1. IF a method with the same name as the computed Getter_Method and zero parameters already exists in the Enclosing_Class, THEN THE Getter_Processor SHALL emit a compile-time error on the annotated field.
2. IF two or more `@Getter`-annotated fields in the same Enclosing_Class produce the same computed getter name, THEN THE Getter_Processor SHALL emit a compile-time error on the conflicting fields.

### Requirement 7: Generated Getter Method Visibility

**User Story:** As a developer, I want the generated getter method to be public, so that the getter is accessible from any context.

#### Acceptance Criteria

1. THE Getter_Processor SHALL generate each Getter_Method with `public` access modifier.

### Requirement 8: Getter Behavior Under Field Hiding in Inheritance

**User Story:** As a developer, I want predictable getter behavior when a Derived_Class hides a field from a Base_Class, so that getter generation respects the `@Getter` annotation placement and accesses the correct field in the class hierarchy.

#### Acceptance Criteria

1. WHEN a Derived_Class hides a field from a Base_Class (Field_Hiding) and only the Base_Class field is annotated with `@Getter`, THE Getter_Processor SHALL generate the Getter_Method only on the Base_Class, and the Derived_Class SHALL inherit the Base_Class Getter_Method (which accesses the Base_Class field).
2. WHEN a Derived_Class hides a field from a Base_Class (Field_Hiding) and only the Derived_Class field is annotated with `@Getter`, THE Getter_Processor SHALL generate the Getter_Method only on the Derived_Class, and the Getter_Method SHALL access the Derived_Class field.
3. WHEN a Derived_Class hides a field from a Base_Class (Field_Hiding) and both the Base_Class field and the Derived_Class field are annotated with `@Getter`, THE Getter_Processor SHALL generate a Getter_Method on the Base_Class that accesses the Base_Class field, and THE Getter_Processor SHALL generate an overriding Getter_Method on the Derived_Class that accesses the Derived_Class field.
4. WHEN a Derived_Class hides a field from a Base_Class (Field_Hiding) and both fields are annotated with `@Getter`, THE Derived_Class Getter_Method SHALL override the Base_Class Getter_Method (same method name, same return type, same zero-parameter signature).
5. WHEN a Derived_Class hides a field from a Base_Class (Field_Hiding) and neither field is annotated with `@Getter`, THE Getter_Processor SHALL generate no Getter_Method for that field name in either class.
6. WHEN a Derived_Class hides a field from a Base_Class (Field_Hiding) and both fields are annotated with `@Getter`, and the Derived_Class field type is a subtype of the Base_Class field type (Covariant_Return_Type is possible), THE Getter_Processor SHALL generate an overriding Getter_Method on the Derived_Class with the Derived_Class field type as the return type.
7. WHEN a Derived_Class hides a field from a Base_Class (Field_Hiding) and both fields are annotated with `@Getter`, and the Derived_Class field type is NOT a subtype of the Base_Class field type (Covariant_Return_Type is not possible), THE Getter_Processor SHALL emit a compile-time error indicating that the getter override is not allowed due to incompatible return types.

### Requirement 9: Generic Field Type Preservation

**User Story:** As a developer, I want the generated getter to preserve the full generic type of the annotated field, so that type safety is maintained without requiring manual casts.

#### Acceptance Criteria

1. WHEN a Generic_Field is annotated with `@Getter`, THE Getter_Processor SHALL generate a Getter_Method whose return type preserves the complete generic type including all type parameters. For example, a field of type `List<String>` produces a getter returning `List<String>`, not `List`.
2. WHEN a Generic_Field with nested type parameters is annotated with `@Getter`, THE Getter_Processor SHALL preserve all levels of type parameterization in the Getter_Method return type. For example, a field of type `Map<String, List<Integer>>` produces a getter returning `Map<String, List<Integer>>`.
3. WHEN a field uses a type variable from the Enclosing_Class (e.g., `T value` in a class `Foo<T>`), THE Getter_Processor SHALL generate a Getter_Method whose return type is the same type variable `T`.

### Requirement 10: Volatile and Transient Field Restriction

**User Story:** As a developer, I want the processor to reject `@Getter` on volatile or transient fields, so that I am warned about generating accessors for fields with special JVM semantics.

#### Acceptance Criteria

1. IF a Volatile_Field is annotated with `@Getter`, THEN THE Getter_Processor SHALL emit a compile-time error indicating that getter generation is not supported for volatile fields.
2. IF a Transient_Field is annotated with `@Getter`, THEN THE Getter_Processor SHALL emit a compile-time error indicating that getter generation is not supported for transient fields.

### Requirement 11: Anonymous Class Restriction

**User Story:** As a developer, I want the processor to reject `@Getter` on fields inside anonymous classes while allowing it in other class kinds, so that getter generation is limited to contexts where the generated method is reachable by name.

#### Acceptance Criteria

1. IF a field annotated with `@Getter` is declared inside an Anonymous_Class, THEN THE Getter_Processor SHALL emit a compile-time error indicating that getter generation is not supported inside anonymous classes.
2. WHEN a field annotated with `@Getter` is declared inside an enum, THE Getter_Processor SHALL generate the Getter_Method normally.
3. WHEN a field annotated with `@Getter` is declared inside a Named_Inner_Class, THE Getter_Processor SHALL generate the Getter_Method normally.
4. WHEN a field annotated with `@Getter` is declared inside a Local_Class, THE Getter_Processor SHALL generate the Getter_Method normally.
5. WHEN a field annotated with `@Getter` is declared inside a Static_Inner_Class, THE Getter_Processor SHALL generate the Getter_Method normally.

### Requirement 12: Inherited Method Collision Detection

**User Story:** As a developer, I want the processor to detect collisions between a generated getter and a method inherited from a superclass, so that I am informed of conflicts at compile time.

#### Acceptance Criteria

1. IF the computed Getter_Method name for a `@Getter`-annotated field matches the name of a zero-parameter Inherited_Method from a superclass (where the Inherited_Method is not generated by the Getter_Processor), THEN THE Getter_Processor SHALL emit a compile-time error on the annotated field indicating a collision with the inherited method.
2. THE Getter_Processor SHALL distinguish between methods inherited from a superclass that were generated by the Getter_Processor (covered by Requirement 8) and methods that were manually declared in the superclass (covered by this requirement).
